# 地域対応 実装設計

交換券価値最適化ツール（単一HTML・ビルド不要）を**地域（エリア）対応**にする。
地域ごとに採取資源・汲める液体・バッテリー・転送可否が異なる。レシピ自体は全地域共通で変えない。
「ある地域で作れない物が多い」のは、**入力（採掘資源・液体）の有無からレシピが自動で使えなくなる結果**として表現する（レシピの明示無効化はしない）。

本書は確定仕様。設計判断は全て確定済み（§11）。実装はコールドスタートのサブエージェントへ委譲可能な粒度で記す。

## 0. 位置づけ・対象ファイル

- 本書は上位設計。**転送の最適選択の内部機構**（候補分類・material/resource経路・solveScenario・Δpt実測・verify省略）は
  既存 `転送システム-実装設計.md` の §3〜§6・§9〜§10 をそのまま流用する。本書はそれを**地域リンクで包む**外側を定義する。
- 既存転送ドキュメントとの差分（本書が上書きする点）:
  - 転送は「谷地固定・枠1」ではなく **(供給元地域→受取地域) の有向リンク**。受取地域は他地域それぞれから1品目ずつ同時受信。
  - `EXTERNAL_MAP` / `resourceLimits` は**選択中の受取地域**＋有効リンクから組み立てる。
  - データは地域固有/共通に分離（§2）。`config.json` を分割。
- 対象: メイン `交換券最適化ツール.html`、データ `data/config.json`・`data/recipes.json`・`data/machines.json`・
  `data/resources.json`（→ `data/regions.json` へ統合）・`data/transfer.json`。
- 着手前に対象HTMLと全 data/*.json を Read して構造把握すること。
- レシピは scanner 自動生成。`recipes.json` は触らない。

## 1. 全体方針

- **1地域フォーカス型＋タブ切替**。タブで受取地域を1つ選び、その地域に対して最適化を回す。
- **N地域へ拡張可能な構造**にする（現状は武陵・四号谷地の2地域。増える想定）。
- 転送は **(供給元→受取) リンク**。受取地域タブで、他地域ごとに転送 ON/OFF できる。
  N地域なら受取地域は最大 N-1 リンク（各リンク1品目）を同時受信。**N=2 の今は1リンク**で現転送設計と等価。

## 2. データモデル（共通 / 地域固有の分離）

ファイル数は不問。可読・手入力しやすさ優先で以下に正規化する。

  ```json
    "selfSupply": ["芽針の種", "サンドリーフの種"],
    "batteries": [
      { "name": "小容量武陵バッテリー", "genPerSec": 1600, "cycleSec": 40 },

      { "name": "小容量谷地バッテリー", "genPerSec": 220,  "cycleSec": 40 },
      { "name": "中容量谷地バッテリー", "genPerSec": 420,  "cycleSec": 40 },
      { "name": "大容量谷地バッテリー", "genPerSec": 1100, "cycleSec": 40 }
    ],
    "mining": {
      "purityRate": { "高": 20, "低": 10 },
      "machinePower": { "電動I": 5, "電動II": 10 },
      "waterPumpPower": 10
    }
  }
  ```
  - `byproductGlobal`: 汚水・不活性壌晶廃液はレシピ副産物由来＝全地域で常時供給扱い。
  - `selfSupply`: 各植物生産（採種機・無入力）に紐づき**地域非依存＝共通**。谷地で芽針が使えないのは
    水が無い地域フィルタ（§5）で自動成立するので、selfSupply 自体は地域で分けない。
  - `batteries`: **共通**。バッテリーは電力源としてどの地域でも消費（発電）できるため genPerUnit は地域非依存。
    ある地域でどの電池を発電に使えるかは「その地域で生産できる or 転送で持ち込んだ」かに依存し**創発的に決まる**
    （武陵に谷地バッテリーを転送して発電に充て、自前バッテリーを交換券へ回す、等が成立）。`genPerUnit` は §3 で派生。

  - 旧 config 直下の `batteryGen`/`byproductSupply`/`selfSupply`/`exchangeItems` 定義は廃止。
    `batteryPriority` も廃止（不要。§2.1 末尾の注記参照）。
  - 旧 `externalSupply` キーは削除。
- `transfer.json`:
  ```json
  {
    "ratePerHour": 1500,
    "items": [
      { "name": "源石鉱物", "value": 1, "producibleIn": ["buryo", "yachi"] },
      { "name": "大容量谷地バッテリー", "value": 20, "producibleIn": ["buryo", "yachi"] }
      /* 現60品目すべて producibleIn:["buryo","yachi"]。今後追加する品目は ["buryo"] のみ */
    ]
  }
  ```
  - `value` は品目固有＝共通の数字（1か所のみ）。地域固有なのは `producibleIn`（どの地域が供給可能か）だけ。
  - 現行の全品目は「谷地から送れる物は武陵からも送れる」前提で両地域。新規は武陵専用の余地を残す。

## 2.1 アイテム→レシピ解決（`recipe` フィールド廃止）

exchangeItems から `recipe` 指定を撤廃し、**アイテム名から「それを出力するレシピ」を引く**。

- 解決関数: `recipeForItem(name)` = `RECIPES` のうち `outputs[].name === name` を満たすレシピ。
- **曖昧性の扱い**: 同名 output を複数レシピが持つ場合（例: 壌晶廃液 は本流レシピと不活性壌晶還元の両方が出力、
  水 は専用ポンプと副産物の両方）、**`byproducts` ではなく `outputs` に主産物として出すレシピを優先**。
  なお現データの exchangeItems（武陵/谷地）は全て主産物が一意なので実害はないが、解決ロジックは曖昧性に備える。
- 影響: 旧 `exchangeItems[].recipe` を参照していた箇所（最終スケール対象レシピの特定）を `recipeForItem(name)` に置換。

### batteryPriority 廃止の注意

`batteryPriority` を撤去する。現行 computeFlow が BATTERY_PRIORITY をバッテリー発電の選択順に使っている場合、
**順序非依存（最適化器/LP が発電に使う電池を選ぶ、または genPerUnit 効率順で自動決定）**へ置換する。
撤去前に computeFlow / 発電収支での BATTERY_PRIORITY 参照箇所を grep すること。
挙動不変の確認は**転送なしベースラインに限る**（転送ありは次項の拡張で挙動が変わるのが正しい）。

**発電在庫の拡張（順序非依存化とは別の必須変更）**: 発電に充てられる電池数は
**「当該地域での生産数 + EXTERNAL_MAP による供給数」の合算**とする。現行は交換品レシピの
ライン数×出力のみが発電在庫であり、EXTERNAL_MAP はレシピ入力の需要削減にしか効かないため、
電池（どのレシピの入力にも現れない）を転送しても誰にも消費されない。この合算を入れないと
§2 の「谷地バッテリーを武陵へ転送して発電に充て、自前バッテリーを交換券へ回す」が成立せず、
転送候補としての電池が常に Δpt=0 になる。

## 3. 導出式と確定値

ロード時に生データから派生値を計算する（生整数＋秒数/純度を保持し、計算済み値は持たない方針）。

```
miningPower = (defaultMax / purityRate[purity]) × 実power
  実power: 電動I/II → machinePower[machine]
           水力      → waterPumpPower / 2   （赤銅2基1セット平均・水力採鉱機本体は電力0）

genPerUnit  = genPerSec × cycleSec / 60      （武陵小: 1600×40/60 = 1066.67）
```

確定値（全資源 purity=高=20/分）:

| 地域 | 資源 | defaultMax | machine | miningPower |
|---|---|---|---|---|
| 武陵 | 源石鉱物 | 540  | 電動I  | 135 |
| 武陵 | 青鉄鉱物 | 90   | 電動II | 45  |
| 武陵 | 赤銅鉱物 | 360  | 水力   | 90  |
| 谷地 | 源石鉱物 | 560  | 電動I  | 140 |
| 谷地 | 青鉄鉱物 | 1080 | 電動II | 540 |
| 谷地 | 紫晶鉱物 | 240  | 電動I  | 60  |

| 地域 | 電池 | genPerSec | genPerUnit |
|---|---|---|---|
| 武陵 | 小 | 1600 | 1066.67 |
| 武陵 | 中 | 3200 | 2133.33 |
| 谷地 | 小 | 220  | 146.67  |

## 4. ロード時処理（initData の再構成）

1. fetch: config / recipes / machines / regions / transfer（旧 resources.json は廃止）。
2. 受取地域を1つ選択（タブの現在値。初期は先頭地域）。
3. 選択地域から実効データを構築:
   - `RESOURCES` = 当該地域の resources（`miningPower` を §3 で派生付与）。
   - `FIXED_POWER` = `RESOURCES` の miningPower 合計 + `fixedPowerOther`（[html:1375] と同形）。
   - `BATTERY_GEN` = **共通 batteries**（`genPerUnit` 派生）の全体マップ。地域では絞らない。
     どの電池を実際に発電へ使えるかは生産可否＋転送（創発的）で決まる。batteryPriority は廃止（§2.1 注記）。
   - `EXCHANGE_ITEMS` = 当該地域 exchangeItems（`recipe` は `recipeForItem` で解決）。
   - 液体供給: `liquidSupply`（地域固有）＋ `byproductGlobal`（全地域）＝この地域で無償供給される素材集合。
   - `selfSupply` = 共通（config）。地域では分けない。
4. ベースラインの `EXTERNAL_MAP` は `{}`（転送なし）。`resourceLimits` は `RESOURCES` の defaultMax。
5. 地域切替（タブ）時は 2〜4 を再実行し、モデル再構築・再最適化。

> 「無償供給される素材集合」を computeFlow / calcLines がどう食うかは現行の externalSupply/byproductSupply 経路に合わせる。
> 旧 `byproductSupply`（汚水・不活性・水・沈殿酸の固定リスト）の役割を、`liquidSupply + byproductGlobal` の地域別合成へ置換する。

## 5. 地域フィルタリング（レシピは無効化しない）

- レシピは共通のまま。ある地域で作れない物は、**必要な採掘資源が地域の resources に無い／必要な液体が
  liquidSupply+byproductGlobal に無い**ことで、`expandChain`/`calcLines` が供給を満たせず自動的に不成立になる。
- 例（谷地）: 水が無い → 芽針・息壌・液化息壌・赤銅塊・芽針エキス系が不成立。赤銅鉱物が無い → 赤銅/緋銅チェーン全滅。
  結果 `緋銅部品`・`芽針注射剤I/II` 等の交換品は谷地で生産0になる（最適化が自然に0を返す）。
- **紫晶鉱物は消費レシピ未登録**のため、谷地で採れても当面は使い道なし（死蔵）。scanner で紫晶チェーン取込後に自動で活きる。
- 交換品（exchangeItems）は**地域固有**。各地域の品目・pt・tags を使う。地域で到達不能（必要レシピ未登録 or
  入力不足）な品は最適化結果が0。表示でグレーアウト等は §8。

## 6. 転送の地域リンク化

- 受取地域 R のタブで、他地域 S ごとに「S→R 転送 ON/OFF」トグルを出す。
- リンク S→R が ON のとき、転送候補 = `transfer.items` のうち `producibleIn ∋ S` かつ R の生産に効く品目。
  実効供給量(/分) = `ratePerHour / value / 60`。
- **何でも送れる**。受取地域で使えれば Δpt として価値が出て、使えなければ Δpt=0。特別な送信制限は設けない
  （`producibleIn` は供給元 S が出せるかの制約のみ。受取側は供給された物の価値を実測するだけ）。
- 候補の種別分類は **受取地域 R 基準で行う**（効かせ方は転送ドキュメント §3・§4 を流用）:
  - `name` が **R の resources に存在** → **resource**: `limits[name] += qty`（採掘上限緩和）。
  - それ以外で `name` がレシピ入力に現れる（電池含む。発電源として消費されうる物も「使える」） → **material**: `EXTERNAL_MAP[name]=qty`。
  - どちらでもない → **none**: Δpt=0。
- ポイント: 同じ資源でも「R で採れる地域」では resource 経路、「R で採れない地域」では直接入力＝material 経路に倒れる。
  分類が R 依存なので、R に無い資源を送っても recipe 入力なら material として効き、入力でもなければ none。これで §12-d が解ける。

## 7. 最適化の外側ループ

- 受取地域 R を固定し、有効リンク集合 L（ON の供給元地域）について最適化する。
- 各リンクは1品目を供給。**N=2 の現状は L が高々1リンク**なので、転送ドキュメント §5 の単一枠ループ
  （ベースライン＋候補ごとに solveScenario→Δpt 比較→最大採用）をそのまま使える。
- **N≥3 で複数リンク同時**になった場合: 各リンクの品目選択は相互干渉するため厳密には組合せ
  （Πリンク候補数）。当面は **リンクごとに貪欲（独立に最良品目を選び、合成して1回 computeFlow で確定）**で近似し、
  必要時に双対値ベースの同時精査へ拡張（§12）。N=2 では組合せ問題は発生しない。
- 採用シナリオ確定後は必ず `computeFlow()` を通して最終 result を作る（転送ドキュメント §9 の `finish` 同様）。

## 8. UI

- **地域タブ**: 受取地域を切替。切替で §4 再構築＋再最適化。
- **転送トグル**: 受取地域以外の各地域に ON/OFF。ON のリンクのみ最適化対象。
- 採用転送の表示: リンクごとに（供給元地域・品目名・実効qty/分・種別・Δpt）を1行。
- 候補別 Δpt ランキング（material/resource を降順、none は「未モデル＝効果測定不可」と別掲）— 転送ドキュメント §8 準拠。
- 資源テーブル: 当該地域の resources のみ表示。resource 転送採用時は「+qty 転送供給」を上限緩和として可視化。
- 地域で到達不能な交換品はグレーアウト or 別掲（生産0の理由が分かるように）。
- 採用シナリオの EXTERNAL_MAP/limits を renderResult 前に global へ確定（転送ドキュメント §8 と同じ順序要件）。
- 旧 `extRows`/`芽針エキス青鉄ボトル` 参照バグ（転送ドキュメント §2・§8）はこの機に除去。

## 9. computeFlow への影響（ベースライン不変の維持）

- `computeFlow(finalScales, resourceLimits)` のシグネチャ・ロジックは維持。受け取る `RESOURCES`・`FIXED_POWER`・
  `BATTERY_GEN` が「選択地域のもの」に差し替わるだけ。
- **武陵を選択した状態のベースライン（転送なし）が現行と数値一致すること**を必ず確認（手順を壊さない）。
  - 注意: 現行 `resources.json` の miningPower は 源石150/青鉄60/赤銅30。本設計の導出値は 135/45/90 で**変わる**。
    → ベースライン一致の基準は「**導出値を入れた武陵**」とする。現行値との差異は仕様変更として許容（§11 で確定済み）。
    現行値との完全一致が必要なら導出を使わず旧値直書きも可だが、本設計は導出を採用。
- miningPower は生産規模に依らない固定オーバーヘッド（現行踏襲）。消費比例化は将来フェーズ（§12）。

## 11. 確定済み設計判断

- スコープ: 1地域フォーカス＋タブ。全地域同時最適化はしない。
- 転送: (供給元→受取) 有向リンク。各リンク1品目。受取側は他地域から1品ずつ同時受信（N=2は1リンク）。
- 共通/固有分離: value は共通、producibleIn が地域固有。resources/液体/**exchangeItems** は地域固有。
  汚水・不活性壌晶廃液は全地域常時供給、水・沈殿酸は地域固有（liquidSupply）。
  **selfSupply・batteries は共通**（selfSupply=植物生産紐付け、batteries=どの地域でも消費できる電力源で genPerUnit 地域非依存）。
- 転送は何でも送れる。価値は受取地域での Δpt 実測で決まる。候補分類は受取地域基準（resource/material/none）。
- exchangeItems の `recipe` フィールド廃止。アイテム名→出力レシピを `recipeForItem` で解決（§2.1）。
- batteryPriority 廃止（発電選択は順序非依存化）。
- miningPower 導出: 全資源 高純度20/分。電動I=5・電動II=10。赤銅=水力（本体0）＋水ポンプ(10)/2＝2基1セット平均。
  → 現行ハードコード値(150/60/30)から導出値(135/45/90)へ変更することを許容。
- genPerUnit 導出: `genPerSec × cycleSec / 60`。生整数＋秒数を保持（非40s電池の将来対応のため）。
- 現転送品目は全て producibleIn=両地域。今後追加は武陵のみ。
- レシピは無効化せず、入力の有無で自動的に作れる/作れないが決まる。

## 12. 残課題・要確認

- (a) N≥3 の複数リンク同時選択の厳密最適化（当面は貪欲近似。双対値プルーニングは必要時）。
- (b) miningPower の消費比例化（`電力 = Σ実消費 × 単位電力`）。computeFlow の電力収支を変えベースライン数値が動くため別フェーズ。
- (c) 谷地 exchangeItems の多く（喬花カプセル・谷地バッテリー・鋼製部品・紫晶系・結晶外殻）は**対応レシピが
  recipes.json 未登録**。scanner で更新予定。取込まではこれらの交換品は生産0（表示は §8 のグレーアウト/別掲で対応）。
  （d は §6 の受取地域基準分類で解決済み: 採れない資源でもレシピ入力なら material として効く）。
